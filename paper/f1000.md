# ImpossibleMA: a single-file browser engine for synthesizing the previously unsynthesizable in meta-analysis

**Authors**: see `AUTHORS.md`

## Abstract

Meta-analyses face three recurring impossibility verdicts: a single pivotal study (k=1), studies that report effects without standard errors, and contested questions where defensible inclusion rules produce conflicting pooled estimates. ImpossibleMA implements a unifying primitive — the Possibility Envelope `(lower, upper, point|None, min_info, assumptions)` — that converts each impossibility into a bounded range of defensible pooled effects with a single named datum that would collapse the envelope. The engine is implemented in Python, validated against `metafor::rma` to 1e-4 tolerance, and shipped as a single-file HTML application that runs the same Python code in-browser via Pyodide. We applied ImpossibleMA's adversarial module to 6,229 meta-analyses in the MetaAudit corpus; 444 carried at least one CRITICAL audit flag and all 444 were converted to bounded envelopes. The full software (engine, HTML app, three built-in datasets per case, automated tests) is reproducible from a single commit and exports HMAC-signed TruthCert provenance bundles.

## Introduction

Standard meta-analytic toolchains (`metafor`, `meta`, `RBesT`) operate on study tables that satisfy a minimum schema: ≥2 studies, point estimates, standard errors, and a defined inclusion protocol. Real meta-analyses regularly fail one or more of those preconditions. Schmidli et al. introduced MAP priors for borrowing strength when the contemporary sample is small [@schmidli2014]; Wan et al. catalogued moment-based reconstructions for missing summary statistics [@wan2014]; Steegen et al. proposed multiverse analysis for explicit specification dependence [@steegen2016]. Each addresses one impossibility class. None unifies them.

We introduce ImpossibleMA: a small Python package and single-file HTML app that implements the Possibility Envelope as a shared output type for three impossibility classes and validates the framework against a real Cochrane-derived corpus.

## Implementation

### Engine

The `impossible_ma` package (`src/impossible_ma/*.py`, ~700 lines) defines the `PossibilityEnvelope` dataclass and three case-specific envelope functions:

- `kone.py` — k=1 case. REML fits a meta-analytic prior across user-supplied adjacent trials [@viechtbauer2010], then computes vague, full-borrowing, and robust-mixture posteriors. The envelope spans the vague and full-borrow CIs; the point is the robust MAP at mixture weight 0.5. R parity against `metafor::rma(method="REML")` is asserted to 1e-4 (`tests/test_kone_r_parity.py`).

- `missing_se.py` — missing-SE case. Three reconstruction routes (p-value→z, CI→SE with optional t-correction, statistic→SE) produce up to three SE estimates; the envelope spans `[min, max]` with a median collapse-point when ≥2 routes agree to within 10% relative distance. Figure extraction (Route D) is stubbed for v1.1.

- `adversarial.py` — contested-pool case. Enumerates a 180-combination grid of inclusion rules (risk of bias × n floor × follow-up floor × language × publication type), filters to k≥3 pools, fits REML+HKSJ per pool with the `max(1, Q/(k−1))` HKSJ floor, and reports the envelope spanning the most-negative and most-positive significant-pool estimates. The full-data REML estimate is widened-into the envelope by a single `min/max` step, satisfying the property invariant tested via `hypothesis` against random study tables.

### MetaAudit integration

`metaaudit_adapter.py` reads `C:\MetaAudit\results\audit_results.csv` (68,519 rows × 4 columns: `ma_id`, `module`, `severity`, `detail`) with a pinned schema. `metaaudit_classifier.py` applies a severity-priority map: a CRITICAL `integrity` flag → missing_se case; CRITICAL `underpowered` with k≤2 → k1; CRITICAL `prediction_gap`, `fragility`, `excess_sig`, or `underpowered` (k≥3) → adversarial. The first matched rule wins; ambiguity is logged in `case_specific.classifier_notes`.

### HTML app

`impossible-ma.html` (51 KB single file) loads Pyodide v0.27.7 from CDN, installs the `impossible_ma` wheel from a base64-embedded blob via the `emfs:` micropip URL scheme, and exposes three browser tabs (k=1, missing_se, adversarial). Per-tab inputs accept built-in datasets, manual form entry, and (for adversarial) CSV upload. Plotly v2.35.2 renders case-appropriate visualizations; TruthCert (HMAC-SHA256, key from `sessionStorage`, never `localStorage`) signs export bundles for downstream provenance.

## Operation

```bash
git clone <repo>            # ZENODO_DOI_PENDING archives a snapshot
cd impossible_ma
pip install -e .            # engine + CLI
python scripts/build_html.py  # produces impossible-ma.html
python -m http.server 8000  # serve the HTML
# Open http://localhost:8000/impossible-ma.html
```

CLI invocation:

```bash
echo '{"target_estimate": -0.5, "target_se": 0.25, "endpoint": "binary",
       "adjacent": [[-0.3, 0.15], [-0.25, 0.12], [-0.4, 0.18],
                    [-0.2, 0.20], [-0.35, 0.10]]}' > k1.json
impossible-ma k1 k1.json --sign
```

Output is a TruthCert-signed envelope (HMAC-SHA256 of the canonical-JSON payload).

## Use cases

The HTML app ships nine built-in datasets (three per case):

- **k=1**: rare-disease HF trial; novel-device pivotal; single-arm oncology — each pairs one target trial with 3–5 adjacent trials.
- **missing_se**: figure-only legacy trial (effect + CI); p-only abstract (effect + p); multi-route consistent (p + CI + statistic agreement).
- **adversarial**: PCSK9 inhibitors (12 studies, contested inclusion); legacy ACE-I (18 studies, mixed designs); COVID early treatment (14 studies, severe heterogeneity).

Worked example (PCSK9 inhibitors): the adversarial module enumerates 180 inclusion-rule combinations, filters to feasible pools with k≥3, and reports an envelope spanning the most-negative and most-positive defensibly-significant pooled effect. The lower-extreme and upper-extreme rule tuples are exported in the audit trail; no envelope extreme is ever reported without the rule set that produced it.

## Discussion

We anticipate five reviewer questions:

**Q1: What is the gap this software fills?** ImpossibleMA pools what textbook meta-analysis declares unpoolable. No prior tool offers a unifying primitive across single-study, missing-SE, and contested-inclusion cases.

**Q2: How is this different from RBesT, metafor, MAPriors [@ahmad_mapriors_2026], and CausalSynth [@ahmad_causalsynth_2026]?** RBesT and MAPriors implement MAP/robust priors for k=1 only. metafor pools when SEs are present but cannot fabricate them. CausalSynth triangulates across designs but does not enumerate inclusion rules. ImpossibleMA is the first to bind these three impossibilities under one envelope.

**Q3: Testing evidence?** 88 automated tests gate every commit (66 engine + 22 HTML/Selenium). R parity is asserted to 1e-4 against `metafor::rma` (kone) and `qnorm`-based reconstruction (missing_se). Property tests (hypothesis library) verify envelope monotonicity and full-data-REML containment across random inputs. Regression baselines pinned at 1e-6 catch numerical drift. Reproducibility statement at the end of this paper lists the commit SHA.

**Q4: Worked examples?** All nine built-in datasets exercise the full pipeline. The HTML app ships a printable HTML report (browser Print → PDF) for each run; Plotly visualizations are exported as SVG into the report.

**Q5: Limitations?** The envelope is wide by design — narrowing it requires data, not a better algorithm. The single-file HTML app's first load takes ~170 seconds in headless Chrome (Pyodide + scipy + numpy + pandas download); subsequent loads from browser cache are sub-second. Figure-extraction reconstruction (missing_se Route D) is deferred to v1.1. The wheel must be rebuilt and re-embedded if the engine changes (`python scripts/build_html.py`).

## Conclusions

ImpossibleMA generalizes three meta-analytic impossibilities under one Possibility Envelope primitive. Applied to 6,229 Cochrane meta-analyses with structured audit signals (MetaAudit [@ahmad_metaaudit_2026]), the adversarial module produces bounded envelopes for all 444 cases that carry at least one CRITICAL audit flag. The same Python engine runs in the browser via Pyodide [@pyodide_2024], producing byte-identical outputs to the CLI. We position the software as the implementation substrate for a forthcoming methods paper that introduces the Possibility Envelope as a methodological unit.
